# 项目审计循环 · 第二轮 Spec

## Why
第一轮审计已修复全部 Schema/列名不一致问题（15 个修复）。但审计深度仅限于数据库 Schema 对齐和部署配置。第二轮需深入检查 TypeScript 类型安全、运行时错误处理、逻辑缺陷、安全性等更深层问题。

## What Changes
- 审计范围：扩展到类型系统、错误处理、API 一致性、安全性、潜在运行时缺陷
- 审计维度：类型检查、错误处理、边界条件、安全漏洞、代码逻辑

## Impact
- Affected specs: `project-audit-cycle`（前一轮审计已完成）
- Affected code: 全部 `server/src/`、`web/src/`、`shared/types/`

## 第二轮审计维度

### 维度 1：TypeScript 类型安全
- 类型导入路径是否可解析（`@shared`、`@shared/index` 等别名）
- 接口/类型定义与实际使用是否一致
- 是否存在 `any` 类型滥用

### 维度 2：错误处理完整性
- API 路由是否有 try-catch 包裹
- 异步操作是否有 `.catch()` 或 try-catch
- 数据库连接失败是否有降级处理
- LLM 调用失败是否有重试/降级

### 维度 3：API 端点一致性
- 前端 API 客户端方法名与后端路由是否匹配
- 请求/响应类型是否对齐
- 状态码使用是否规范

### 维度 4：安全性
- SQL 注入防护（参数化查询覆盖率）
- 认证中间件覆盖范围
- 敏感信息是否可能泄露

### 维度 5：代码逻辑
- 边界条件处理
- 空值/undefined 检查
- 竞态条件
