## 1. 设计哲学与核心原则

v5.0 承袭 v4.8 的全部原则，并在以下方向进一步凸显：

**认知共生**：人类与 AI 不再是单向的“审阅与被审阅”。通过 AI 问答、证据溯源、静默观察和纠错反哺，知识库成为一个持续对话、共同进化的认知场域。每一次问答既是获取知识的行为，也是知识库自我完善的契机。

**全汉化与无障碍**：从系统底层消息到用户界面，从提示词到证据原文翻译，实现统一的汉语环境。汉语不是一层翻译外衣，而是系统思考的原生语言。

**长期可维护**：版本归档防止文件膨胀，幽灵关系清理保持图谱健康，影子评估异常熔断防止系统性污染。一个运行数年的知识库应当像一座精心保养的图书馆。

**可传世与去中心化**：Markdown 作为唯一且完全自包含的真相源，使得知识库本身不依赖任何特定运行时。静态站点生成接口确保其可脱离服务端独立存在，成为可被任意设备浏览的数字遗产。

**人类掌权不可动摇**：所有的自动化都是辅助。State 的修改永远需要人类确认，全自动写入永远可回滚，AI 的建议永远以 Diff 形式呈现。效率的提升绝不牺牲人类对知识的最终裁判权。

#### v4.1→v4.2：分级 Diff 审核（解决确认疲劳）

v4.0 的"100%人类确认"确保了安全，但引入了规模化瓶颈——"确认疲劳"。当知识库积累数千实体时，每天审核 20 个 Diff 成为心理负担。

**解法**：分级 Diff 审核。所有写入仍需人类点击（保留掌权原则），但按置信度 × 影响范围三级分流，提供不同的操作粒度：批量折叠、列表预览、逐个标红。

| 级别     | 触发条件                                             | UI 呈现               | 操作粒度               | 目标               |
| -------- | ---------------------------------------------------- | --------------------- | ---------------------- | ------------------ |
| 🟢 批量区 | 置信度 > 0.9 且 低影响（错别字、别名补充、格式统一） | 自动折叠，仅显示摘要  | 一键批量合并全部       | 消除噪声，10秒清空 |
| 🟡 预览区 | 置信度 0.7-0.9 或 属性/关系更新                      | 列表展示 + [预览影响] | 逐条预览后合并/拒绝    | 知情决策           |
| 🔴 重点区 | 置信度 < 0.7 或 矛盾/高风险                          | 醒目标红，强制逐个    | 必须逐个确认，无法批量 | 聚焦真正需要判断的 |

即使是🟢批量区，也不会自动写入——仍需用户主动点击"全部合并"按钮。系统永远不自动写入 State。分级的是操作效率，不是自动化程度。

#### v4.2→v4.3：三项基础强化

- **知识图书馆**：建立源文件档案库（`library/objects/`），通过实体锚点与 wiki、图谱深度融合，实现证据链闭环。

- **语境感知矛盾检测**：引入语境标签，区分"冲突"与"多义"，同一语境冲突才仲裁，不同语境定义并存为 `[context_variant]`。

### 2.2 全汉化策略（界面与系统消息）

为统一交互语言、降低认知负担，全系统采用**默认为纯汉语**的全汉化方案，并预留国际化扩展能力。

**前端界面**：
- 所有 UI 文本（按钮、菜单、提示、占位符、错误信息）抽取至 `locales/zh-CN.json`，使用 `react-i18next` 库渲染。
- 同时预留 `locales/en.json` 等文件，支持一键切换语言包，但默认仅加载汉语资源。
- Web App 首次加载时根据浏览器语言偏好自动选择，但优先使用用户配置。

**系统提示词**：
- AI Agent 编排层的 Planner、Generator、Grader、Reflector 的系统提示词全部模板化，存放于 `skills/prompts/` 目录。
- 每个模板文件提供 `zh-CN` 版本作为主模板，英文版作为备选（如 `planner.zh-CN.md` 和 `planner.en.md`）。
- 运行时根据 `.brain-config.yml` 中的 `language` 字段自动选择，默认 `zh-CN`。

**错误与日志**：
- 所有运行时错误消息、审计日志、CLI 输出、夜间简报均默认为汉语。
- 内部异常堆栈保留原始技术信息，便于调试，但面向用户的消息强制本地化。
- 日志存储格式包含 `lang` 字段，便于多语言环境下检索。

**知识内容**：
- 所有 `wiki/` 条目、`summaries/` 报告、`raw/` 人类笔记默认采用汉语书写。
- 图谱节点标签、时间线条目、Diff 建议文本、证据悬停提示也均输出为汉语。
- LLM 生成的 State、Assessment、语义年轮等，通过提示词约束输出为汉语。

**证据双语呈现**（v5.0 新增）：
- 当 `evidence_span` 的源语言与界面语言（汉语）不一致时，系统在生成回答时异步调用低成本翻译模型将证据原文翻译为汉语。
- 翻译结果缓存于 `evidence_translations` 表，下次引用时直接复用。
- 前端脚注弹出层默认显示汉语译文，原文折叠，用户可展开查看双语对照。
- 这保证了全汉化体验，同时不丢失溯源精度。

**配置**：
- `.brain-config.yml` 中新增 `language: zh-CN` 字段，全局生效。
- 同时在配置中可指定 `fallback_language: en`，以备翻译失败时使用。

此策略确保整个知识库从系统层到内容层形成统一的汉语环境，降低使用门槛，同时不关闭国际化可能。

### 4.3 L1 · AI Agent 编排层

**标准循环**：Planner → Retriever → Grader → Generator，由 Reflector 控制迭代。

**L1.1 标准 Agentic RAG 循环**

- **Planner**：分析问题或任务，制定检索计划和推理步骤。
- **Retriever**：调用 L2 混合检索引擎，获取相关文档片段和实体。
- **Grader**：使用评分卡评估检索质量，包含以下维度：
  1. 事实准确（1-5）：声明是否与提供的证据一致，有无幻觉
  2. 覆盖完整（1-5）：是否回答了用户问题的所有方面
  3. 来源清晰（1-5）：每个声明的溯源是否明确
  4. **证据覆盖率**（0-1）：输出中有来源引用的声明占全部声明的比例
- **Generator**：基于检索结果和评分反馈生成答案。每个事实性声明必须附带 `evidence_span`，或显式标记为 `[inferred]`。`evidence_span` 包含标准化文本中的字符偏移和原始文件位置的双向映射。
- **Reflector**：评估是否需要补充检索。将"已检索但未被引用的证据片段"列表注入下一次 Generator 调用的提示，明确要求基于缺失证据进行补充。

**可控反思机制**：
- **信息增益追踪**：记录每一轮新引用的实体/证据数量。如果新轮次新增实体和证据均 <1 条，Reflector 自动停止并返回当前最优结果。
- **预算熔断**：硬上限——最多 5 轮反思，总耗时不得超过 3 秒（T2 延迟约束）。任一条件触发即终止，返回当前结果并记录终止原因到日志。
- **停止策略优先级**：连续两轮评分不提升 > 信息增益为零 > 熔断触发。

**L1.2 共生问答专属策略**

**追问压缩**：
- 当同一对话轮次超过压缩阈值（默认 5 轮），Reflector 在下一轮 Planner 之前，调用低成本模型（如 Haiku）对前几轮问答进行摘要压缩。
- 压缩格式："用户询问了关于 X 的问题，系统回答的核心是 Y，引用了 Z 等来源。用户随后追问了 A，系统解释了 B...（共 N 轮历史）"
- 压缩后的摘要作为"对话上下文"注入下一轮提示，替代完整历史。
- 用户可在界面看到"对话已压缩"的提示，并可选择展开完整历史或临时关闭压缩。

**静默观察与补提取**：
- 每次问答后，系统记录被引用但状态为 `partially_extracted` 或 `new` 的文件到 `observed_files` 表，递增 `reference_count`。
- 同一文件在观察期内被引用次数达到触发阈值（默认 3 次）后，夜间任务自动对该文件执行 `extract_facts`，生成 🟡 预览 Diff。
- 若用户在观察期间通过"纠错"或"有帮助"按钮给出正向反馈，可立即提前触发提取。
- 观察列表和计数值可通过 `BrainAPI.listObservedFiles()` 查询。

**纠错反哺自动化**：
- 用户在共生问答面板点击"回答有误"按钮，系统弹出高亮文本选择框，用户标记错误陈述。
- 系统自动将该错误陈述与对应的 `evidence_span` 原文片段打包为反例（`source_text` + 错误陈述），存入 `shadow_benchmarks`。
- 同时标记源文件状态回退为 `partially_extracted`，在 Timeline 追加："用户纠错：某陈述被标记为错误，文件待重新提取"。
- 无需人类提供正确事实，完全自动化采样。

**证据翻译缓存**：
- 在生成回答的同时，若 `enableTranslation` 为 true 且证据源语言非汉语，后台异步调用低成本翻译模型。
- 翻译结果缓存于 `evidence_translations` 表，键为 `(span_id, target_lang)`。
- 下次同样证据被引用时直接复用缓存，不重复翻译。缓存过期时间 90 天。

**L1.3 问答反思终止策略**

- 信息增益为零：连续两轮未新增实体或证据，停止。
- 预算熔断：最多 3 轮反思，总耗时 < 5 秒。
- 证据覆盖率达标：Grader 评分中"证据覆盖率" ≥ 0.8 且置信度稳定，停止。

**L1.4 新增工具**

- `narrate`：时间线叙事生成，接受实体 slug 或集群 ID，按时间线拼合 State 变更、Library 文件时间、版本历史，生成概念演化故事。纯读，不写入存储。
- `shadow_eval`：在沙箱环境中运行基准事实库，返回回归报告（已知错误复现率、新增遗漏、正确率变化）。
- `rule_learn`：从用户确认操作中学习实体映射规则，写入本地 user_rules 规则库。
- `ask_question`：共生问答入口，支持多轮对话、证据翻译、追问压缩。

### 4.4 L2 · 混合检索引擎 + 轻量 NLI 校验

**全文搜索统一**：
- 放弃 SQLite FTS5，统一使用 **Postgres 内置全文搜索**（`tsvector` / `tsquery`）。
- PGLite（Postgres 的 WASM 移植）同样完整支持 `tsvector`，确保本地开发与生产环境一致。
- Library 文件的全文索引（PDF 文本、OCR 文字、音频转写）统一纳入 `page_fts` 表，列类型为 `tsvector`。

**组件清单**：

| 组件                | 来源                        | 说明                                                      |
| ------------------- | --------------------------- | --------------------------------------------------------- |
| 向量搜索            | pgvector HNSW               | 开源，零自研                                              |
| 全文搜索            | PG tsvector/tsquery         | 数据库内置，统一替代 FTS5                                 |
| RRF 融合            | 自研 (~50行)                | 轻量融合算法，融合向量和全文排序结果                      |
| 图谱遍历            | Postgres CTE                | 纯 SQL，利用现有 links 表递归查询                         |
| 重排序              | zerank-2                    | ZeroEntropy API，默认开启，可配置关闭                     |
| 来源感知            | 自研 (配置驱动)             | 来源可信度加权，不同来源类型赋予不同基础权重              |
| 命名实体 + 学习路由 | 自研 (正则 + user_rules 表) | 用户确认即学习，规则持久化到本地规则库                    |
| 轻量 NLI 校验       | RoBERTa-mnli (开源)         | 矛盾检测前置，仅在蕴含"矛盾"或"中立"时进入 LLM            |
| 意图路由            | 自研分类器 (5 类)           | 具体事实 / 全局主题 / 跨领域综合 / 原始文件搜索 / AI 问答 |

**NLI 校验流程**：
1. 矛盾检测任务被触发时，先用 RoBERTa-mnli 对两条陈述进行"蕴含/矛盾/中立"三分类。
2. 如果判断为"蕴含"，则不报警，不进入 LLM 分析。
3. 如果判断为"矛盾"或"中立"，则送入 LLM 进行深度语境分析，并决定是否生成 🔴 重点区矛盾 Diff 或 🟡 预览区语境建议。
4. 所有 NLI 判断结果缓存在 `nli_cache` 表中，键为 (陈述A hash, 陈述B hash)，避免重复计算。

**三层响应延迟**：
- T0 < 50ms：pgvector + tsvector + RRF 融合，返回快速初筛结果
- T1 < 250ms：T0 + 图谱 CTE + zerank-2 + 来源感知 + 命名实体，返回完整混合检索结果
- T2 < 2s：T1 + Agentic 多轮（含可控反思），返回深度综合回答

**查询分层策略**（v5.0 扩展）：
- 具体事实 → T0/T1 混合检索
- 全局主题 → 预生成社区报告（夜间更新，直接读取 summaries/）
- 跨领域综合 → T2 Agentic 多轮
- 原始文件搜索 → Library 全文索引 + 元数据搜索
- AI 问答 → 意图路由后选择 T0/T1/T2，用户可指定模式

#### 4.5.3 版本历史归档策略

**问题**：高频更新实体的 `## Version History` 会无限膨胀，影响 Markdown 可读性和解析性能。

**v5.0 解决方案**：

**归档规则**：
- 当某实体的活跃 `## Version History` 条目数超过**活跃阈值**（默认 50 条）时，夜间维护任务自动触发归档。
- 归档过程：
  1. 取最早的 N-20 条（保留最新 20 条在 Markdown 中），生成一条**归档摘要**，格式如："2025-01 至 2026-06 期间共 87 次更新，关键转折：v23 引入信息论视角，v45 合并物理与信息定义，v72 加入生物学语境"。
  2. 将详细历史条目完整移入 `changelog/` 目录下的专用文件，如 `changelog/entropy.md`，每个条目保留完整的时间戳、变更内容和溯源信息。
  3. 在原 Markdown `## Version History` 区块底部，用归档链接替代已移除的条目："> 较早的 87 条版本历史已归档至 [[changelog/entropy]]"

**归档保证**：
- 归档文件同样采用标准 Markdown 格式，包含完整的 `## Version History` 区块。
- `rebuild-struct` 命令会同时解析 `wiki/` 和 `changelog/` 中的版本历史，完整重建 `knowledge_versions` 表。
- 归档操作视为全自动写入，记录在 `auto_change_log` 中（操作者为 system，批次号包含 `archive-` 前缀）。
- 支持回滚：回滚归档批次时，系统将归档文件中的详细历史重新合并回原 Markdown，然后执行 `rebuild-struct`。

**摘要生成**：
- 使用低成本模型（Haiku）扫描归档的版本历史条目。
- 识别变更频率和类型（新增 State、修改 Assessment、合并实体、新增关系等）。
- 识别关键转折点（语境新增、定义变更、矛盾解决）。
- 生成 2-3 句话的汉语摘要，标注关键版本号和变更类型。

#### 4.5.4 幽灵关系清理

**问题**：当人类删除或重命名 `wiki/concepts/old-theory.md` 时，通过 `Relations` 指向它的其他实体中会残留死链，形成"幽灵关系"。

**v5.0 解决方案**：

**检测时机**：
- `rebuild-struct` 执行时，扫描所有 `Relations` 中的 `[[wikilink]]`。
- 夜间 `lint` 步骤中，同样执行此检查。

**检测逻辑**：
- 对每个 `[[wikilink]]`，检查目标文件是否存在于文件系统。
- 若目标不存在，触发清理流程。

**清理流程**：
1. 在源实体的 `## Open Threads` 中自动追加一条："- [ ] 关系 [[已删除实体]] 的目标不存在，请手动清理或确认删除"
2. 生成 🟢 批量区 Diff，建议删除该无效关系行。
3. 在 DB 的 `links` 表中，将对应边标记为 `orphaned`，不计入正常图谱遍历。
4. `ghost_relations` 表中记录详情：源实体 slug、目标实体名称、发现时间、状态。
5. 健康仪表盘"幽灵关系数"指标更新。

**用户操作**：
- 用户可在 🟢 批量区一键合并清理建议，删除无效关系行。
- 也可在仪表盘点击幽灵关系卡片，查看详情并逐条处理。
- 若用户后来恢复了被删除的文件，对应的 `ghost_relations` 记录自动标记为 `resolved`。

#### 4.6.2 全局预算管理器

**设计目标**：防止夜间自动任务或频繁问答导致 API 费用失控。

**实现**：
- 用户可在 Web 仪表盘或 CLI 设置：
  - `daily_budget`：日预算上限（默认 $5.00）
  - `monthly_budget`：月预算上限（默认 $50.00）
  - `max_single_ask_cost`：单次问答最大费用（默认 $0.50）
- BrainAPI 每次调用 LLM 前，通过 `getRemainingBudget()` 检查剩余预算。
- 对于**用户主动触发的操作**（手动查询、手动提取、手动触发维护、AI 问答）：不受全局日夜预算限制。
- 对于**夜间自动 Dream Cycle 任务**：在开始每一阶段前检查。如果当日已消费金额超过日预算，或当月已超月预算，则**跳过所有需要 LLM 调用的任务**，仅执行白名单规则、lint、索引更新、幽灵清理等免费操作。
- 对于**AI 问答**：虽然不受全局预算熔断限制，但每次调用会检查单次最大费用。若预估费用超过上限，则降低模式（从 detailed 降为 concise）或限制检索范围。每日问答总消费单独统计。

**预算消耗追踪**：
- 每次 LLM 调用记录实际 token 消耗和预估费用，按任务类型分类（提取、年轮、矛盾分析、问答等）。
- 仪表盘按类别展示当日和当月消费明细。

**仪表盘显示**：
- "今日 API 费用"数值（已消费 / 日预算），按类别分色块
- "本月 API 费用"进度条
- "今日问答消费"单独计数
- 如果夜间预算熔断，仪表盘顶部显示橙色横幅："夜间自动任务已暂停。可提高上限或等待明日重置。"
- "按当前消耗速率，月预算预计可支撑 X 天"

**批量预检闸门**（进一步减少不必要 LLM 调用）：
- 该实体过去一周无新 Timeline 条目、无 State 变更 → 跳过语义年轮生成
- 集群成员数无变化，且无新文件加入 → 跳过缺口分析
- 文件已标记为 `fully_extracted` 且无再提取信号 → 跳过事实提取

#### 4.6.3 交叉验证增强

- **文件哈希去重**：只有来自不同 `library/` 原始文件（SHA-256 不同）的提取事实才计为独立来源。
- **作者/出版物去重**：如果元数据可用，同一作者的不同著作计为半独立来源（权重 0.05 而非 0.1）。
- **交叉验证分数公式**：
  ```
  cv_score = min(1.0, 0.7 + 0.1 × distinct_file_sources + 0.05 × semi_distinct_sources)
  ```
- 孤证（仅一个来源支持）最大 cv_score = 0.7。多源支撑可以逐步提升至 1.0。

#### 4.6.4 全自动写入回滚机制

**设计**：
- 每次全自动写入操作（批量）生成一条 `auto_change_log` 记录，包含：
  - `batch_id`：唯一批次标识
  - `timestamp`：执行时间
  - `affected_files`：修改的 Markdown 文件路径列表
  - `change_summary`：变更摘要（如"追加 3 条版本历史，更新 2 个年轮，归档 1 个实体"）
  - `pre_change_snapshot`：变更前的文件内容备份（短期存储于 `.brain/snapshots/`）
- 健康仪表盘的"全自动变更面板"列出最近 24 小时的所有批次，归档操作以 `archive-` 前缀区分。
- 用户可选择一个批次，点击"回滚"：
  1. 系统从快照中恢复该批次影响的所有 Markdown 文件（包括归档文件恢复）。
  2. 自动执行 `brain rebuild-struct`，基于恢复后的 Markdown 重建所有索引、向量、图谱、全文搜索。
  3. 回滚操作记录审计日志。

#### 4.6.5 触发机制

**实时（全自动）**：
- 新文件写入 `raw/` 或 `library/objects/` → `auto_link` 提取 wikilink 关系 + `embed` 生成向量 + 文件状态设为 `new`
- 新页面创建 → `extract_facts` 运行（低成本模型），生成 Timeline 条目和 State Diff 候选
- Timeline 追加：所有提取事件、文件状态变更、再提取信号、问答日志自动追加
- 白名单规则引擎：格式统一、别名补充、错别字修正直接写入
- 文件状态更新：`new` → `partially_extracted` → `fully_extracted`
- 再提取标记：人类修改 State 后，自动标记关联源文件为 `partially_extracted`
- 纠错反哺标记：用户纠错后，标记源文件为 `partially_extracted`，追加反例
- `/maintain` 命令：手动触发完整维护管线

**夜间（全自动 + Diff 建议）**：
1. **预算检查** → 若已达日/月上限，跳过所有 LLM 任务，仅执行免费操作（白名单、lint、幽灵清理、索引更新），记录日志，结束夜间流程。
2. `community_detect` + `community_report`：社区检测和报告生成（基于图谱结构，无需 LLM）
3. NLI 预检 → 语境感知 `contradict_detect`：先 NLI 后 LLM 深度分析
4. `forget_decay`：遗忘衰减计算
5. `lint` + `fix_citations`：格式检查与修复
6. **幽灵关系清理**：检测死链，生成清理建议 Diff，标记 orphaned
7. `topic_cluster`：聚类算法运行 → 生成 `summaries/` 集群报告（全自动写入）
8. `cluster_gap_analysis`：分析集群知识缺口（强推理模型）
9. `enrich_external`：外部知识补充（如 API 查询）
10. State/Assess Diff 生成：为已有实体生成改进建议（含 evidence_span）
11. 语义年轮生成：检测版本链中显著变更次数 ≥3 的实体，生成新年轮（强推理模型）
12. **版本历史归档检查与执行**：遍历实体，若活跃版本条目 > 50，执行归档并生成摘要
13. **证据翻译缓存刷新**：删除 90 天无访问的翻译缓存
14. 全自动写入历史区块：追加 Version History 和 Semantic Rings Archive

**每周**：
- `skillopt`：Fat Skills 自我优化
- `shadow_eval`：影子评估回归测试（含异常熔断检查）

### 4.7 L4.5 · 影子评估引擎 (Shadow Eval) + 异常熔断

#### 4.7.1 反例定义

- 反例是一个 `(源文本片段, 被人类拒绝的 AI 错误提取陈述)` 对。
- **评估目标**：给定源文本，模型是否**不再**输出该已知错误陈述。
- **生成方式**：
  - 自动采样：当人类在 Diff 审核中拒绝一条 State 建议时，系统自动将该建议的 `evidence_span` 对应的源文本片段和该被拒绝的陈述打包为一个反例。
  - 问答纠错采样（v5.0 新增）：当用户在共生问答面板点击"回答有误"并标记错误陈述时，系统自动打包为反例。
  - 两种方式均存入 `shadow_benchmarks` 表的 `negative_examples` 列。
- **回归测试**：将源文本输入提取管线（或问答生成管线），检查输出中是否包含被禁错误陈述。如果包含，标记为"已知错误复现"，扣分。人类无需额外提供任何"正确事实"。

#### 4.7.2 正例来源

- 从人类确认且 ≥90 天未修改的 State 中自动采样，标记为 `trusted`。
- 正例评估维度：管线输出是否仍能提取出与 Golden Answer 语义一致的事实。

#### 4.7.3 版本关联

- 影子评估报告关联当前 `skills/` 目录的 Git commit hash。
- 每次修改 Skill 文件后，强制运行一次影子评估。

#### 4.7.4 异常熔断机制（v5.0 新增）

**触发条件**：每次 `shadow_eval` 运行后，对比本次与上次评估的指标变化。满足以下任一条件则触发异常熔断：

- 已知错误复现数波动 > 200%（相对于上次，且绝对数量 > 10）
- 正例遗漏数波动 > 300%
- 新增错误数 > 50 且上次为 0

**熔断行为**：
- 系统自动中止本次评估，保留上次正常评估结果。
- 在健康仪表盘和影子评估报告中生成 🔴 重点区告警：
  > "影子评估结果出现异常波动，可能由模型变更、管线故障或反例污染引起。请人工检查以下可能原因后重新运行：1) 是否更换了底层模型？2) 提取管线是否正常？3) 反例库是否需要去重清理？"
- 异常事件记录在 `eval_anomaly_flags` 表中，包含时间、波动指标、上次值、本次值。
- 异常熔断**不阻止其他夜间任务**，仅阻止本次评估结果覆盖上次正常结果。
- 用户可通过 CLI `brain shadow-eval --force` 强制执行，跳过熔断检查。

#### 4.7.5 报告指标

- 已知错误复现数 / 总反例数
- 正例遗漏数 / 总正例数
- 新增错误数（不在反例库中的新幻觉）
- 与上次评估相比的正确率变化
- **异常熔断状态**（正常 / 已熔断 / 人工确认后恢复）

### 4.9 L5.5 · 知识图书馆 (Library) + 媒体服务 API

**目录结构与安全**：

```
library/                      ← 整体加入 .gitignore，不被 Git 跟踪
├── .library-index.db          ← SQLite 数据库 (文件元数据索引)
├── by-category/               ← 按分类树的符号链接，指向 objects/
├── by-date/                   ← 按摄入日期归档，符号链接指向 objects/
└── objects/                   ← SHA-256 命名的原始文件，内容寻址去重
```

**文件状态机**：

```
new ──→ partially_extracted ──→ fully_extracted ──→ superseded
  ↑             │                                      │
  └─────────────┘ (再提取反馈回退、纠错反哺回退)
```

- `new`：刚摄入，尚未提取事实。
- `partially_extracted`：至少完成一次提取，但可能存在被忽略的线索或人类纠错反哺。
- `fully_extracted`：已充分提取，短期内无需重新处理。
- `superseded`：已被更新版本替代，归档但保留溯源链。
- 状态变化自动写入相关实体的 Timeline，并在图谱文件节点颜色上体现（灰色=未处理，蓝色=部分，绿色=完成，虚线=被替代）。

**证据链接双向映射**：
- `evidence_span` 存储两种引用：
  - `source_text_offset`：在标准化 Markdown 中的字符偏移（稳定，不依赖原始格式）
  - `original_location`：原始页码（PDF）或时间码（音视频），用于 UI 反查原始文件
- 摄入管道保留 MinerU 输出的页码信息和 Whisper.cpp 的时间戳。
- UI 点击 source 链接时，通过 `original_location` 定位到原始 PDF 页面或音频秒数。
- 问答脚注点击时，弹出双语证据（优先展示汉语译文，可切换原文）。

**再提取反馈环（结构化差异 + 反例）**：
- 人类修改 State 后，系统将"原 LLM 提取事实"与"人类修改后事实"的 Diff 传入下次提取的 LLM 上下文。
- 被人类删除的错误事实缓存为反例，按实体类型分组，后续同类型提取时作为 few-shot 反例注入提示。
- 问答纠错产生的反例同样进入此反馈环。

**敏感信息防护**：
- `.brain-config.yml` 中配置 `sensitive_patterns`（身份证号、手机号、邮箱等正则表达式）。
- 提取管道在生成 `evidence_span` 时对匹配模式自动脱敏（替换为 `[已脱敏]`）并记录日志。
- 问答生成答案后，同样经过脱敏检测，确保已脱敏内容不外泄。

**媒体服务 API**：
- `GET /api/library/:hash`：通过 BrainAPI 暴露，支持 `Range` 请求（HTTP 206），实现视频拖动。
- 鉴权：与 L0.5 认证集成，阻止未授权访问。
- 缓存：`Cache-Control` 头设置长期缓存，基于内容哈希的 URL 天然支持永久缓存。

### 4.10 L6 · 多模态摄入管道 (BrainIngest)

**支持的输入模态**：
- 📄 文档 (PDF/DOCX/PPTX/XLSX) → MinerU → Markdown (公式→LaTeX, 表格→HTML)
- 📷 图片 (JPG/PNG/WebP/GIF/BMP/TIFF) → MinerU(OCR) + VLM(描述) → 文字 + 语义描述
- 🎵 音频 (MP3/WAV/M4A/OGG) → Whisper.cpp → 带时间码的文字转录 (95%+准确率)
- 🎬 视频 (MP4/MOV/AVI) → FFmpeg 提取音轨 → Whisper.cpp 转录 → 带时间戳的文字
- 🌐 网页 (URL/HTML) → Trafilatura → 干净 Markdown (去广告/导航)
- 📝 文本 (MD/TXT/CSV/JSON) → 直接通过

**处理步骤**：
1. **模态转换**：将各类输入统一转为 Markdown
2. **内容清洗**：去广告、去页眉页脚、去重复段落、格式化统一
3. **证据双向映射建立**：记录标准化文本中的字符偏移与原始页码/时间码的映射关系
4. **原始文件存档**：保留原始文件到 `library/objects/`，计算 SHA-256 哈希
5. **实体提取与关联**：LLM 提取实体并生成 evidence_span
6. **文件状态初始化**：设置为 `new`，准备进入 Dream Cycle

**远期增强**：图片区域标注（Phase 6），VLM 对图片特定区域进行标注提取为实体，溯源指向图片坐标。

#### 4.11.1 媒体原地渲染详细设计

**设计目标**：在 Web App 浏览 Wiki 页面时，`library://` 链接自动转换为可交互的图片、视频、音频，无需离开页面或手动下载。同时 Markdown 源文件保持标准链接语法，对 Git 友好。

**Markdown 语法保持不变**：
- 图片：`![熵的热力学定义示意图](library://sha256-img-001)`
- 视频：`![熵讲座片段](library://sha256-video-001#t=45s)`
- 音频：`[会议录音](library://sha256-audio-001#t=120s)`

**markdown-it 自定义渲染规则**：
1. 检测链接以 `library://` 开头，提取哈希和片段参数（`#t=...` 或 `#page=...`）。
2. 查询 `.library-index.db` 获取文件的 MIME 类型。
3. 根据 MIME 类型：
   - `image/*`：输出 `<img src="/api/library/{hash}" loading="lazy" />`，支持懒加载。
   - `video/*`：输出 `<brain-media type="video" src="/api/library/{hash}" start="{seconds}" title="{描述}"></brain-media>`
   - `audio/*`：输出 `<brain-media type="audio" src="/api/library/{hash}" start="{seconds}" title="{描述}"></brain-media>`
   - 其他（PDF 等）：渲染为带图标的可点击链接，点击调用文件预览组件。
4. 媒体上方或下方保留一个小引用角标，悬停时显示完整 `library://` 链接和溯源信息。

**`<brain-media>` Web Component**：
- 属性：`type` (video/audio)，`src` (API 路径)，`start` (秒数)，`title` (描述)。
- 功能：
  - 加载后自动跳转到 `start` 指定时间。
  - 简洁控制栏（播放/暂停、时间显示、音量、全屏）。
  - 右侧引用角标，点击复制 `library://` 链接。
- 实现：基于 Custom Elements API，框架无关。

**安全与性能**：
- 媒体 API `/api/library/:hash` 要求认证。
- 基于内容哈希的 URL 支持永久缓存。
- 支持 HTTP Range 请求，视频可随意拖动进度条。

## 5. 认知共生：AI 问答与知识反哺

本章为 v5.0 核心新增章节，完整阐述 AI 问答如何与知识库融为一体，实现双向进化。

### 5.1 问答执行流程

当用户通过任何入口（Web App、MCP、CLI）提出问题时，系统执行以下流程：

**第一步：意图路由**
- 复用 L2 意图分类器，将问题分类为：具体事实 / 全局主题 / 跨领域综合 / 原始文件搜索。
- 分类结果决定检索策略和反思深度。

**第二步：多路检索**
- 具体事实类：T0/T1 混合检索（向量 + 全文 + 图谱）。
- 全局主题类：优先读取预生成的 `summaries/` 社区/集群报告，辅以 T1 检索。
- 跨领域综合类：升级至 T2 Agentic 多轮检索。
- 文件搜索类：Library 全文索引 + 元数据检索。

**第三步：证据锚定生成**
- 调用强推理模型（Sonnet/GPT-4o）生成 Markdown 格式回答。
- 每个事实性声明强制附带 `[^span_id]` 脚注，链接到具体的 `evidence_span`。
- 无法溯源但合理的推断标记为 `[inferred]`。
- Grader 评估证据覆盖率。若覆盖率 < 0.6，触发反思补充。

**第四步：双语证据准备**（并行异步）
- 若 `enableTranslation` 为 true（默认），检查每个 `evidence_span` 的源语言。
- 若源语言非汉语，查询 `evidence_translations` 缓存。
- 缓存未命中则调用低成本翻译模型（Haiku），翻译结果存入缓存。
- 用户可见的回答中不直接显示原文，仅在脚注弹出层中提供双语切换。

**第五步：反思与补充**
- Reflector 检查信息增益。若有未被引用的高相关证据片段，注入下一轮 Generator 提示。
- 最多 3 轮反思，总耗时 < 5 秒。
- 证据覆盖率达标（≥ 0.8）或连续两轮评分不提升则提前终止。

**第六步：返回结果**
- 返回 `AskResponse`：Markdown 答案、来源列表（含译文）、关联实体、对话 ID、token 消耗、预估费用。
- 若引用了处于静默观察的文件，`observationTriggered` 为 true。
- 若进行了追问压缩，`compressedHistory` 为 true。

### 5.2 追问压缩策略

**触发条件**：
- 当同一 `conversationId` 的对话轮次超过 `compressionThreshold`（默认 5 轮）时，在下一轮问答前自动触发压缩。

**压缩执行**：
- Reflector 调用低成本模型（Haiku）扫描前几轮的完整问答记录。
- 生成一句压缩摘要，格式如下：
  > "对话历史摘要：用户首先询问了关于 [主题A] 的问题，系统引用 [N] 个来源进行了解答。用户随后追问了 [子问题B]，系统进一步解释了 [要点C]。用户接着对 [声明D] 提出了质疑...（共 [X] 轮对话）"

- 压缩摘要替代完整历史，作为"对话上下文"注入下一轮的 Planner 提示中。

**用户体验**：
- 在共生问答面板顶部，显示 "对话已压缩（显示最近 5 轮详情）" 的状态提示。
- 用户可点击"展开完整历史"查看之前的所有问答。
- 用户可在提问时追加特殊指令 `[完整上下文]` 来临时禁用压缩。

**成本节约**：
- 压缩后，每轮问答的上下文 token 消耗从线性增长变为恒定（仅压缩摘要 + 最近 5 轮）。
- 对于长达 20 轮以上的深度对话，token 成本可降低 60-80%。

### 5.3 静默观察与补提取

**观察机制**：
- 每次问答后，系统遍历答案中引用的所有 `evidence_span`，提取对应的 Library 文件哈希。
- 若文件状态为 `new` 或 `partially_extracted`，在 `observed_files` 表中递增 `reference_count`。
- `first_referenced_at` 记录首次引用时间，`last_referenced_at` 更新为当前时间。

**触发提取**：
- 当同一文件的 `reference_count` 达到触发阈值（默认 3 次）时，夜间 Dream Cycle 的 Maintain 阶段自动对该文件执行 `extract_facts`。
- 提取结果生成 🟡 预览区 Diff，进入审核队列。
- 提取完成后，`observed_files` 中该文件的记录标记为 `extracted`，并从活跃观察列表中移除。

**用户加速触发**：
- 用户在问答中对答案点击"有帮助"，系统将该答案引用的所有观察文件的 `reference_count` 直接提升至触发阈值，在下一个维护周期优先处理。
- 用户可在补提取观察列表中手动点击"立即提取"，绕过阈值直接触发。

**防止过度扰动**：
- 同一文件在提取后的 7 天内，即使再次被引用，也不会重新进入观察列表（冷却期）。
- 用户在观察列表中看到的是真正"被多次需要但尚未提取"的文件，避免噪声。

### 5.4 纠错反哺与反例自动采样

**用户纠错流程**：
1. 用户在共生问答面板中，对某条回答点击"回答有误"按钮。
2. 系统高亮回答中的事实性声明（有 `[^span_id]` 脚注的部分），用户可选择标记为错误的陈述。
3. 用户可选填纠错说明（非必填），提交反馈。

**系统自动处理**：
1. 将被标记的错误陈述与对应的 `evidence_span` 原文片段打包，生成新反例：
   ```
   {
     source_text: "原文片段...",
     negative_statement: "被标记为错误的 AI 陈述",
     source: "问答纠错",
     created_at: "2026-07-03T10:30:00Z"
   }
   ```
2. 存入 `shadow_benchmarks` 的 `negative_examples` 表。
3. 标记源 Library 文件状态回退为 `partially_extracted`。
4. 在对应实体的 Timeline 追加："🗣 用户纠错：'被标记为错误的 AI 陈述' 被用户指正，源文件标记为待重新提取"。
5. 无需人类提供"正确事实"——反例仅用于检测模型是否复现已知错误。

**反馈闭环**：
- 下次对该文件执行 `extract_facts` 时，该反例作为 few-shot 负面示例注入提示："以下是之前被人类纠正过的错误提取，请不要生成类似的陈述：..."
- 下次影子评估回归时，该反例纳入测试集，检测模型是否复现错误。

### 5.5 证据双语呈现

**翻译触发**：
- 在问答生成阶段，若 `enableTranslation` 为 true（默认），系统检查每个被引用的 `evidence_span` 的源语言。
- 源语言检测使用轻量语言识别库（如 fasttext/langdetect）。

**翻译执行**：
- 调用低成本翻译模型（如 Claude Haiku 或 GPT-4o-mini），提示词指定为汉语。
- 翻译结果与原文一并存入 `evidence_translations` 表：`(span_id, source_text, translated_text, source_lang, target_lang, model, created_at)`。
- 缓存有效期 90 天，过期后在下一次被引用时重新翻译。

**前端展示**：
- 共生问答面板的脚注弹出层，默认展示汉语译文。
- 译文下方有一行小字："原文 (English) ▼"，点击展开查看原文。
- 在 Wiki 页面的 `evidence_span` 悬停框中，同样展示双语（若译文存在）。
- 证据翻译指示器（"译"图标）显示在脚注旁边，表示该证据有可用译文。

**手动翻译**：
- 用户可通过 CLI `brain translate-evidence --all` 批量翻译所有未翻译的证据。
- 或通过 MCP 工具 `translate_evidence` 指定 span 列表。

### 5.7 安全与成本控制

**预算策略**：
- AI 问答为用户主动触发，不受全局夜间预算熔断限制。
- 但每次问答消耗的 token 和预估费用实时统计，并在仪表盘单独展示"今日问答消费"。
- 单次调用受 `max_single_ask_cost` 限制（默认 $0.50）。若预估费用超过上限，系统自动降级策略：
  - 从 `detailed` 模式降为 `concise` 模式
  - 限制检索范围（减少反思轮次）
  - 若仍超预算，提示用户："此问题预估费用较高（$X.XX），是否继续？"

**证据强制**：
- 若 Generator 无法提供满足最低证据覆盖率（0.6）的回答，系统明确回复：
  > "当前知识库中未找到足够证据来回答这个问题。建议补充相关源文件，或尝试用更具体的方式提问。"
- 绝不胡编乱造。

**敏感信息过滤**：
- 回答生成后，复用 `.brain-config.yml` 中的 `sensitive_patterns` 正则进行检查。
- 若匹配到敏感模式，自动替换为 `[已脱敏]`，并记录审计日志。

## 6. 混合检索与查询分层

**组件复用**：
- 向量搜索：pgvector HNSW（开源）
- 全文搜索：PG tsvector/tsquery（数据库内置）
- RRF 融合：自研轻量算法 (~50行)
- 图谱遍历：Postgres CTE 递归查询
- 重排序：zerank-2（ZeroEntropy API）
- 来源感知排序：自研配置驱动
- 命名实体识别 + 学习路由：自研正则 + user_rules 表
- 轻量 NLI 校验：RoBERTa-mnli（开源）
- 意图路由：自研分类器（5 类，新增"AI 问答"）

**三层响应延迟**：
- T0 < 50ms：pgvector + tsvector + RRF 融合，返回快速初筛结果
- T1 < 250ms：T0 + 图谱 CTE + zerank-2 + 来源感知 + 命名实体，返回完整混合检索结果
- T2 < 2s：T1 + Agentic 多轮（含可控反思），返回深度综合回答

**查询分层策略（v5.0 扩展）**：
- 具体事实 → T0/T1 混合检索
- 全局主题 → 预生成社区报告（夜间更新，存储于 summaries/）
- 跨领域综合 → T2 Agentic 多轮
- 原始文件搜索 → Library 全文索引 + 元数据检索
- AI 问答 → 意图路由后自动选择最优层级，用户也可通过 `mode` 参数指定

## 7. 自主维护：Dream Cycle 与长期维护策略

### 7.1 触发机制

**实时（全自动）**：
- 新文件写入 `raw/` 或 `library/objects/` → `auto_link` + `embed` + 文件状态设为 `new`
- 新页面创建 → `extract_facts` 运行，生成 Timeline 和 State Diff 候选
- Timeline 追加：提取事件、文件状态变更、再提取信号、**问答日志**（"🗣 问答：问题摘要 → 引用来源数"）
- 白名单规则自动修正
- 文件状态更新：`new` → `partially_extracted` → `fully_extracted`
- 再提取标记：人类修改 State 后标记源文件为 `partially_extracted`
- **纠错反哺标记**：用户纠错后标记源文件为 `partially_extracted`，追加反例
- `/maintain` 命令 → 完整管线

**夜间（全自动 + Diff 建议）**：
1. **预算检查** → 若已达日/月上限，跳过所有 LLM 任务，仅执行免费操作（白名单、lint、幽灵清理、索引更新），记录日志。
2. `community_detect` + `community_report`（基于图谱结构，无需 LLM）
3. NLI 预检 → 语境感知 `contradict_detect`
4. `forget_decay` + `lint` + `fix_citations`
5. **幽灵关系清理**：检测死链，生成 🟢 批量区 Diff，在 DB 标记 orphaned

## 8. 进化护栏与安全模型

### 8.1 十条护栏

| #   | 护栏                  | v5.0 机制                                                                                                                                                                                                            |
| --- | --------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ①   | 来源锚定              | Diff 必须附带 `library://` 溯源链接 + evidence_span（双向映射：标准化文本偏移 + 原始位置）。无 evidence_span 的 Diff 自动降级置信度并标红。问答答案中每个声明强制脚注溯源                                            |
| ②   | 置信度衰减 + 交叉验证 | 来源(1.0) → 提取(0.9) → 推断(0.7)。cross_validation_score 通过文件哈希去重和作者/出版物去重计算，孤证最大 0.7。分数辅助分级审核决策                                                                                  |
| ③   | 回滚机制 (一致性保证) | 所有变更（含全自动写入和版本归档）记录在 knowledge_versions 或 auto_change_log 中。回滚时恢复 Markdown 文件到变更前快照，然后自动执行 `rebuild-struct` 重建所有索引，保证文件与缓存严格一致                          |
| ④   | 语境感知矛盾          | NLI 预检（RoBERTa-mnli）→ 同一语境冲突 → 🔴 仲裁 Diff；不同语境定义差异 → 并存为 `[context_variant]`。无法裁决时允许标记为 `[contradicted]` 并自动开启 Open Thread 调查                                               |
| ⑤   | 渐进式建议            | 每次 Dream Cycle 最多生成 20 个 Diff。白名单自动修正不限数量。聚类标签全自动。问答补提取 Diff 单次最多 5 个                                                                                                          |
| ⑥   | 分级审核 + 原文对比   | 🟢 批量区（置信度>0.9 + 低影响）：一键批量合并，含幽灵清理建议。🟡 预览区（置信度0.7-0.9 或中等影响）：逐条预览合并/拒绝，展示 evidence_span（双语）。🔴 重点区（置信度<0.7 或高风险/同语境矛盾）：强制逐个确认，无批量 |
| ⑦   | 审计追踪              | 所有写入操作记录：操作者（人类/system/问答纠错）、时间戳、变更内容、变更原因。支持按实体、时间、操作者筛选                                                                                                           |
| ⑧   | 持续学习安全          | 用户确认即学习：实体映射、别名规则写入本地 `user_rules`。反例缓存为只读，不自动修改已有 State。问答纠错自动采样反例，频率驱动的自动白名单建议需人类一键确认后才生效                                                  |
| ⑨   | 安全认证 + 脱敏       | API Key 认证（Bearer Token），Phase 1 即实现。`sensitive_patterns` 配置正则，提取管道和问答输出自动脱敏。`library/` 整体 gitignored                                                                                  |
| ⑩   | 评估异常熔断          | 影子评估波动超阈值（已知错误复现 >200% 或正例遗漏 >300% 或新增错误激增）自动中止评估并告警，保留上次正常结果。防止系统性错误污染评估基准                                                                             |

### 8.2 分级 Diff 审核详细规则

| 级别     | 触发条件                                                          | 示例                                                                          | 操作粒度                             |
| -------- | ----------------------------------------------------------------- | ----------------------------------------------------------------------------- | ------------------------------------ |
| 🟢 批量区 | 置信度>0.9 + cv_score 高 + 低影响                                 | 补充语境标签、格式统一、错别字修正、建议生成年轮、断链提醒、幽灵关系清理建议  | 一键批量合并全部，10秒清空           |
| 🟡 预览区 | 置信度0.7-0.9 或 属性/关系更新                                    | 聚类归属建议、文件状态通知、实体合并建议、分类建议、属性值更新、补提取新事实  | 逐条预览后合并/拒绝，可预览影响范围  |
| 🔴 重点区 | 置信度<0.7 或 同语境矛盾/孤证高风险/无 evidence_span/异常熔断告警 | 矛盾裁决、实体合并冲突、关键属性变更、缺失原文引用的 Diff、评估异常需人工确认 | 强制逐个确认，无法批量，必须人工判断 |

### 8.3 回滚一致性保证

**全自动写入回滚**：
1. 用户在"全自动变更面板"中选择一个批次（含归档批次），点击"回滚"。
2. 系统从 `.brain/snapshots/` 中读取 `pre_change_snapshot`，恢复该批次影响的所有 Markdown 文件（包括从 `changelog/` 恢复归档记录）。
3. 自动执行 `brain rebuild-struct`，基于恢复后的 Markdown 重建所有索引、向量、图谱、全文搜索、版本链。
4. 回滚操作记录审计日志。

**手动操作回滚**：
- 所有人类确认的 Diff 写入也记录在 `knowledge_versions` 表中。
- 用户可以通过 Git 回退 Markdown 文件，然后执行 `rebuild-struct` 完成回滚。
- 对于全自动写入的 Timeline、Version History、年轮等，用户可以通过全自动变更面板的"回滚"按钮精确回滚。

## 附录 B：使用 Markdown 构建 Wiki 百科的实践范式

你可以利用 v5.0 架构轻松搭建一个个人 Wiki 百科，类似维基百科但完全离线、AI 驱动、私密可控。以下是实践指导。

### B.1 目录组织
遵循维基百科的分类传统，在 `wiki/` 下建立清晰的层级：
```
wiki/
├── index.md                  ← 首页，类似维基百科的 Main Page
├── portals/                  ← 主题门户（如"科学"、"历史"、"技术"）
│   ├── science.md
│   ├── history.md
│   └── technology.md
├── categories/               ← 分类页（手动维护或由 AI 生成）
│   ├── physics.md
│   ├── machine-learning.md
│   └── world-war-ii.md
├── concepts/                 ← 具体概念条目
│   ├── entropy.md
│   ├── quantum_mechanics.md
│   └── transformer_architecture.md
├── people/                   ← 人物条目
│   ├── alan-turing.md
│   └── marie-curie.md
├── events/                   ← 事件条目
│   └── moon-landing.md
└── ...
```

### B.2 首页设计 (index.md)
类似维基百科，首页包含：
- **精选条目 (Featured Article)**：手动指定一个条目作为展示，或由系统根据最近修改频率和完整性自动推荐。
- **新闻动态 (Current Events)**：从 Timeline 中自动提取最近 7 天的重要事件（含问答动态），生成一个简洁列表。
- **知识领域导航**：链接到各个 portal 页面，形成一个分类树。
- **搜索框**：直接调用全局检索，输入关键词即可搜索 Wiki 条目、Library 文件和**直接提问**。

首页 Markdown 示例：
```markdown
---
type: index
title: 我的知识百科
---

# 欢迎来到我的知识百科

## 精选条目
本周推荐：[[concepts/entropy]] —— 熵的跨学科理解。

## 知识领域
- [[portals/science]]：物理学、化学、生物学...
- [[portals/technology]]：人工智能、计算机科学...
- [[portals/history]]：古代史、近代史...

## 最近动态
<!-- 系统全自动追加的动态列表，含问答日志 -->
- 2026-07-03 | 🗣 问答：用户询问"量子计算与熵的关系" → AI 回答，引用 3 个来源
- 2026-07-03 | 新增条目：[[concepts/quantum_computing]]
- 2026-07-02 | 更新：[[people/alan-turing]] 语义年轮
```

### B.3 分类与导航
- 每个条目通过 Relations 链接到所属分类，例如：
  ```markdown
  instance_of → [[categories/physics]]
  subfield_of → [[portals/science]]
  ```
- 分类页面可以手动编写，也可以让 AI 通过集群报告自动生成一个"自动分类索引"，存入 `summaries/` 并链入分类页面。
- 图谱可视化提供全局导航，在 Web App 中可从任意条目跳转到相关概念。

### B.4 跨语言/多语境支持
- 利用概念锚点的多语境机制，同一个 `canonical_slug` 可以有多个语言的 State，例如：
  ```markdown
  ### [中文] [extracted]
  熵是系统无序度的量度。

  ### [English] [extracted]
  Entropy is a measure of the disorder of a system.
  ```
- 通过 Relations 链接到语言分类页面，如 `language → [[categories/chinese]]`。

### B.5 质量评级
- 在 frontmatter 中增加 `quality` 字段（stub, start, C, B, A, featured），人类手动评定。
- 集群缺口分析可提示哪些条目质量较低（如缺少来源、只有 stub 定义），自动生成改进建议进入 🟡 预览区。

### B.6 贡献模式（多用户场景）
- 将来 Phase 4 支持多用户时，其他用户可以通过 Diff 审核机制提交编辑建议，由管理员合并，完全复刻维基百科的协作模式。
- 所有用户提交的编辑都走 Diff 审核，保证质量。

### B.7 导出与分享
- 可以将整个 `wiki/` 目录导出为静态 HTML 站点（通过 `brain generate-static-site`），形成一个完整的离线 Wiki。
- Library 文件可选择是否一起打包。
- 静态站点包含完整的图谱可视化（vis-network）、媒体播放器和双语证据展示。

### B.8 AI 问答与 Wiki 的融合
- Wiki 首页和每个条目页面内嵌"Ask AI"入口，用户可就当前条目提问。
- 问答日志自动记录在条目 Timeline 中，成为知识演化的一部分。
- 通过问答发现的缺失知识，通过补提取自动纳入 Wiki。

通过这套范式，你的知识库不再是一堆零散的笔记，而是一个结构严谨、链接丰富、多媒体支持、AI 持续维护与对话的个人百科全书。Markdown 的极简与富媒体的沉浸在这里完美统一。

## 附录 C：静态站点生成器接口预留规范

### C.1 功能定位
静态站点生成器将整个知识库（wiki、summaries、版本归档、图谱）编译为一组纯静态的 HTML/CSS/JS 文件，可脱离 BrainAPI 服务端独立浏览。

### C.2 输入
- `wiki/`、`summaries/`、`changelog/` 目录中的所有 Markdown 文件
- `library/objects/` 中被引用的媒体文件
- `.brain-config.yml` 中的站点配置（标题、语言、主题色等）

### C.3 输出结构
```
exports/static-site/
├── index.html                ← 渲染后的 Wiki 首页
├── wiki/                     ← 渲染后的 wiki 条目 HTML
│   ├── concepts/
│   │   └── entropy.html
│   └── ...
├── summaries/                ← 渲染后的社区/集群报告
├── changelog/                ← 渲染后的版本归档
├── media/                    ← 复制被引用的 library 文件
│   ├── sha256-abc.pdf
│   └── sha256-video-001.mp4
├── assets/
│   ├── css/
│   ├── js/
│   │   ├── vis-network.min.js  ← 图谱静态库
│   │   └── brain-media.min.js  ← 媒体播放组件（独立版本）
│   └── search-index.json       ← 静态搜索索引
└── graph.json                   ← 图谱数据（节点+边）
```

### C.4 转换规则
- **Markdown → HTML**：使用 markdown-it 渲染，所有 `library://` 链接替换为 `./media/` 相对路径。
- **`<brain-media>` 组件**：编译为独立的 Web Component，内嵌于 HTML。
- **图谱可视化**：使用 vis-network 替代 Cytoscape.js，读取 `graph.json` 渲染。
- **搜索**：使用客户端全文搜索（如 MiniSearch 或 Lunr.js），索引预编译为 `search-index.json`。
- **问答功能**：静态站点不含 AI 问答能力，但保留问答日志的展示。
- **证据双语**：若存在翻译缓存，在静态站点中硬编码双语数据为 HTML data 属性。

### C.5 使用方式
```bash
# 导出完整静态站点
brain generate-static-site ./output

# 导出但不包含 library 媒体文件（仅文本和知识结构）
brain generate-static-site ./output --exclude-media

# 导出指定主题门户的子站点
brain generate-static-site ./output --scope=portals/science
```

### C.6 传世性保证
- 生成的站点为纯静态文件，可存放于 U 盘、光盘、静态托管服务（GitHub Pages、Cloudflare Pages）。
- 不依赖任何后端服务、数据库或 API 密钥。
- 媒体文件基于内容哈希命名，永久可寻址。
- 图谱和搜索功能基于客户端 JavaScript，无需服务器计算。
- 整个知识库成为一份真正的数字遗产。
